1. 目标与背景

平台现有核心能力是训练数据标注和采集。随着企业从“训练底层模型”转向“部署可执行任务的 Agent”，数据需求也从孤立样本扩展到任务、环境、工具调用轨迹、偏好对战、失败分类和可验证 verifier。Agent Arena 的核心产品机会，是把真实 Agent 在业务场景中的失败暴露出来，并将失败转化为可销售、可持续迭代的训练数据集。

本 demo 的目标不是做一个展示页，而是让专业用户完整体验：部署/构建 Agent、专家提交难例、沙盒竞技、排行榜奖金、问题报告、解决方案推荐、Appen 数据集采购短名单。

参考对象关键观察对本平台的启发
THU-ZJAI Real-World AI Source README 展示了 Real-World AI Arena、Best Practice Champion、HITL、Task Set 等概念，强调真实业务源、专家实践与任务集沉淀。 Agent Arena 不应只做排行榜；需要把专家、难例、最佳实践和任务集商品化打通。
CREAO Community 面向社区的 AI/Agent 竞赛与榜单形态，强调用户参与、任务挑战、排行和社区激励。 适合采用赛季制、奖励池、公开榜与隐藏集，吸引开发者和专家持续供给任务。
AgentBench 提出用多环境、多任务评估 LLM as Agent，包括网页、游戏、工具使用等能力。 评测维度应覆盖规划、工具调用、环境交互、长期任务和失败恢复。
WebArena / OSWorld 强调真实网页环境、真实电脑环境中的开放式任务，关注可执行动作和环境状态。 沙盒要记录环境状态、动作轨迹、工具结果和验证证据，而非只看最终文本。
GAIA / τ-bench / SWE-bench 分别强调通用助手、多轮工具代理和代码修复真实任务，均需要可验证结果或人工审计。 平台应支持程序化 verifier、专家 rubric、偏好对战和人工复核混合评测。
Chatbot Arena 人类偏好对战可产生可解释排名和市场认知，但并不直接证明任务执行成功。 偏好对战可用于非确定性任务，但要和自动 verifier、隐藏集审计组合。
Appen 官方 Agentic AI / Training Data 页面 Appen 已有 AI training data、agentic task、trajectory、failure taxonomy、RAG evaluation、RL environment 等相关叙事。 Arena 可成为 Appen 数据产品前置诊断入口，直接导向任务集、轨迹集、偏好集和失败分类服务。

来源与不确定性

  • 来源类型：GitHub 项目 README、公开社区站点、公开 benchmark 论文/项目页、Appen 官方产品叙事页面。
  • 确定性较高：Agent 评测需要真实环境、可执行任务、轨迹记录、隐藏集和人工/程序化混合评测；这些在多个 benchmark 和评测平台中重复出现。
  • 产品推断：把 Arena 失败诊断映射为 Appen 训练数据 SKU 是基于 Appen 数据业务能力与 Agent 评测缺口的商业设计推断，需要通过客户访谈和销售 pilot 验证。
  • 需复核项：CREAO 社区的具体赛季机制、外部项目最新功能、开源许可、引用素材可商用性，以及任何真实客户数据进入任务池的授权边界。

3. 用户与 Jobs To Be Done

企业 AI 负责人

核心任务

在采购或自研 Agent 前，希望通过真实业务难例验证效果、成本、安全和可落地性。

  • 对比多个 Agent 的场景排名。
  • 获得可解释的问题分析报告。
  • 确认需要采购哪些训练数据。
Agent 开发者

核心任务

把自有 Agent 部署进沙盒，用难例集定位失败，并通过排行榜获得曝光与奖金。

  • 快速接入 API、镜像或平台模板。
  • 查看轨迹级失败证据。
  • 获得下一步优化建议。
众包专家

核心任务

提交真实难例、金标轨迹和评价 rubric，按质量获得奖励。

  • 提交任务、环境、预期结果和失败点。
  • 维护高价值隐藏集。
  • 参与人工复核和偏好对战。

4. 需求范围

4.1 必须覆盖的基础场景

  1. 支持客户部署自有 Agent 或构建新 Agent，在云端/沙盒环境运行多个场景的难例任务 PK 或评测，并根据排名获得奖金。
  2. 支持众包专家用户有奖提供难例任务，平台审核后进入公开集、隐藏集或数据产品候选池。
  3. 评估不同 Agent 在实际任务中暴露出的多方面问题，结合多个难例任务输出问题分析报告和解决方案。
  4. 引导用户采购 Appen 面向底层大模型/Agent 的训练数据集，包括任务集、轨迹集、偏好集、失败分类和 verifier。
  5. 通过 Agent 竞技面向用户推荐不同场景下优秀的 Agent 解决方案。

4.2 补充设计需求

模块需求成功标准
沙盒接入支持 API Endpoint、容器镜像、云端托管、客户私有 VPC。健康检查、权限检查、工具清单校验可视化。
任务市场支持任务规格、环境、金标答案、金标轨迹、verifier、rubric、奖励。专家可提交并进入审核队列。
竞技评测支持多 Agent、多任务包、公开集/隐藏集、程序化和人工评测。输出实时日志、分项分、排行榜和奖金。
诊断报告按失败分类聚合轨迹，给出根因、影响、修复路径。报告能直接映射到数据产品采购建议。
方案推荐按场景、约束、排名、成本和安全策略推荐 Agent。用户能从“业务问题”走到“Agent 选型”。
商业转化把诊断缺口转化为 Appen 数据产品短名单。客户能看到清晰的数据包、价值和采购入口。

5. 核心指标

指标类别指标解释
供给有效难例通过率、专家复购率、隐藏集污染率衡量任务市场是否能持续供给高质量任务。
评测任务成功率、工具调用正确率、安全合规率、成本效率、人工复核一致性衡量排行榜是否可信。
商业数据产品短名单数、报告转商机率、训练数据采购金额衡量 Arena 到 Appen 数据业务的转化。
用户体验接入时长、首次完成评测时长、报告可行动性评分衡量专业用户是否可以快速获得价值。

6. 非目标与边界

  • Demo 阶段不接入真实客户数据，所有数据均为 mock。
  • Demo 阶段不实现真实沙盒隔离、容器调度和支付发奖，仅展示完整用户体验与产品逻辑。
  • 排行榜仅用于说明评测框架，不代表真实 Agent 能力。
  • 文档中的外部项目参考仅用于产品设计启发，商业化前需做法律与许可复核。
