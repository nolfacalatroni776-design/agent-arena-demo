核心判断

不要做“排行榜网站”，要做 Agent 可信执行能力的操作系统。

未来两年，Agent 测评的主战场会从静态 benchmark 迁移到“任务 + 环境 + 轨迹 + 人类专家 + 生产流量 + 训练数据闭环”。单次打分只能证明某个版本在某套题上表现不错，企业真正需要的是持续回答四个问题：哪个 Agent 能完成我的业务、为什么失败、如何修复、需要补什么数据。

建议定位 Agent Eval OS = 独立可信评测层 + 真实任务环境层 + 轨迹诊断层 + 专家数据生产层 + Agent/数据采购决策层。
5 平台必须同时覆盖的闭环：准入、运行、评分、诊断、改进。
3 市场正在合流的能力：数据生产、工程化 eval、真实环境 benchmark。
2028 终态不再是“谁分高”，而是“谁在某个行业任务里可托付”。

高层 Takeaways

从公开资料看，Abaka 和 Appen 代表“环境与数据生产商”正在前移到 Agent 训练和评测；LangSmith、Braintrust、Galileo 代表“开发者 eval/observability”正在后移到生产监控；Scale 和 LMArena 代表“可信排名”正在解决私有题库、防污染和人类偏好问题。

1. 环境是新数据

任务不再只是 prompt

Agent 能力需要在状态可变化的系统中验证。Abaka 的 RL Environment 页面已经把 API、数据库、数据种子、GUI、MCP-compatible interface 写进产品交付。

2. 轨迹是核心资产

失败发生在过程里

企业不会只买“最终答案对错”，而会买 action sequence、tool result、state delta、failure taxonomy 和 golden trajectory。

3. 信任来自治理

排行榜必须可审计

Scale SEAL 的私有题库、专家生成、透明方法和防过拟合机制，说明“可信评测”本身会成为可销售的产品能力。

4. 最终变成采购入口

评测报告要导向行动

报告终点不是“模型得分”，而是 Agent 选型、数据包采购、prompt/tool 改造、风险 gate 和下一轮回归。

竞品与相邻公司格局

当前市场不是一个单点品类，而是三类能力正在合并：数据与环境供给、持续评测工程平台、公开/私有 benchmark 与 Arena。未来平台要把三者合到同一工作流。

类别 代表对象 公开可验证能力 对未来平台的启发 主要短板或机会
数据 + 环境 Abaka AI 官网展示 Datasets、Abaka Forge Platform、RL Environment、Model Evaluation；RL Environment 强调生产级环境、真实企业仿真、状态化模拟、dense reward、deterministic reset、全栈仿真、MCP-compatible interfaces、六级复杂度任务、judgment-based 与 deterministic 两类评测。 Agent 测评平台必须把“环境构建”变成一等产品对象，而不是在后台临时写脚本。 公开资料更像服务/平台能力展示，独立第三方排名、连续生产监控、Agent 解决方案市场化仍有想象空间。
数据 + 评测 Appen Agentic AI 页面明确包含 Task & Verifier Design、Trajectory Analysis & Failure Mode Taxonomy、Golden Trajectory Creation、Full RL Environment Design、Enterprise RAG Evaluation、SWE-driven workflows。 评测产生的失败证据可以直接转化为训练数据 SKU，这是数据平台切入 Agent 时代的主商业闭环。 如果只做服务交付，难形成高频 SaaS 使用和跨客户可复用的评测资产网络。
可信排行 Scale SEAL / Scale Evaluation SEAL Leaderboards 强调私有数据集、专家评测、防 overfitting；Scale Evaluation 页面强调 proprietary evaluation sets、expert human raters、reporting consistency、targeted evaluations。 平台可信度来自隐藏集、专家治理、方法公开、版本隔离和置信区间，而不是 UI 上的总分。 公开排行更适合模型比较；企业 Agent 还需要客户私有工具、权限、业务政策和生产流量闭环。
持续评测 LangSmith 官方页面定位为 LLM & AI Agent evaluation platform，覆盖 offline/online evals、human feedback、prompt optimization、trajectory/tool call/reasoning 捕获、CI/CD 集成、LLM-as-judge 校准。 未来平台必须支持开发期、上线前、生产期和回归期的连续评测，而不是一次性项目评估。 更偏工程团队自助，缺少行业专家任务供给、评测环境资产市场和训练数据交付闭环。
持续评测 Braintrust / Galileo Braintrust 官方描述强调 tracing production、running evals、catching regressions；Galileo 官方描述强调 evaluate、monitor、protect GenAI applications and agents at enterprise scale。 “可观测性 + eval”会成为 Agent 平台标配。Trace lake 和 regression gate 是基础设施，不是附加模块。 若没有私有任务环境和专家标注网络，难覆盖高价值真实业务难例。
偏好 Arena LMArena / Chatbot Arena 基于匿名随机对战和用户投票形成 Elo/leaderboard，已扩展到文本、图像、视觉等。 开放式任务需要人类偏好信号，但 Agent 执行类任务不能只靠偏好投票，必须结合状态 verifier。 偏好高不等于业务成功。企业场景需要权限、安全、成本、合规和可追溯证据。
真实环境 benchmark WebArena / OSWorld / tau-bench / SWE-bench WebArena 强调 realistic web environment；OSWorld 强调 real computer environment、execution-based evaluation；tau-bench 强调 tool-agent-user interaction、API tools 和 policy guidelines；SWE-bench 代表真实代码修复任务。 平台要支持 Web、桌面、API、多轮用户、代码仓库、RAG、企业系统等多种环境族。 学术 benchmark 通常不直接解决企业私有任务、数据授权、专家运营和商业转化。

2028 年中产品终态

终态平台不是一个“评测页面”，而是企业 Agent 生命周期里的可信控制面。它知道业务目标、环境状态、工具权限、执行轨迹、失败根因、数据缺口和供应商选择。

North Star

每个企业任务都有可复现的 Agent 试验场

用户可以把一个业务流程转成任务包、环境种子、工具权限、verifier、golden trajectory 和风险政策，并对任意 Agent 版本持续回归。

Trust Layer

每个排行榜都有证据链和置信边界

榜单必须展示隐藏集比例、样本量、置信区间、任务族覆盖、人工一致性、泄露风险和版本声明。

Revenue Layer

每份报告都能转化为改进动作

报告自动生成四类行动：换哪个 Agent、补什么数据、改哪个工具/权限、在哪个 production gate 阻断。

未来两年用户场景

面向终态设计时，不应只考虑开发者跑 benchmark。真正高价值用户来自企业 AI 负责人、Agent 开发团队、领域专家、数据交付团队和模型/Agent 供应商。

企业 AI 负责人

采购与上线 gate

  • 比较 3-8 个 Agent 在私有业务场景中的真实表现。
  • 获得可追溯风险报告和上线阈值。
  • 确认是否需要购买数据包或专家评测服务。
Agent 开发团队

调试与回归

  • 接入 API、容器、MCP server、浏览器或桌面 Agent。
  • 查看失败轨迹、工具参数、状态差异和 verifier 结果。
  • 在 CI/CD 中设置 regression gate。
领域专家

难例与金标生产

  • 提交真实难例、政策规则、golden trajectory 和 rubric。
  • 参与冲突仲裁和 LLM judge 校准。
  • 按任务价值、复核质量和隐藏集贡献获得激励。

详细需求分析

需求应围绕“可托付执行”而不是“模型分数”。以下按平台模块列出终态需求，适合后续拆 PRD 和任务。

模块 终态需求 关键对象 成功标准
Agent Registry 支持 API endpoint、container image、hosted agent、browser agent、desktop agent、MCP server、私有 VPC 接入；管理版本、能力声明、工具权限、预算、风险等级。 Agent、Version、Tool Manifest、Credential、Permission Policy 任一 Agent 接入后可通过健康检查、权限检查、工具 schema 检查和 dry run。
Task & Environment Studio 把业务流程转成任务 DSL，包含 instruction、environment seed、database state、GUI state、allowed tools、policy、verifier、rubric、gold trajectory、hidden level。 Task、Environment、Verifier、Rubric、Policy、Seed、Gold Trace 同一任务可 deterministic reset，跨 Agent 复现，且能区分任务失败和环境失败。
Arena Orchestrator 支持公开集、隐藏集、客户私有集、生产抽样集、红队集；支持赛季制、批量回归、A/B、canary、nightly run 和 CI/CD gate。 Arena Run、Season、Task Pack、Sample Strategy、Run Budget 对同一 Agent 版本输出可复现结果，支持置信区间和任务族分榜。
Trace Lake 记录每一步 observation、thought 摘要、action、tool call、tool result、state delta、screenshot、video、token、latency、cost、policy event。 Trace、Step、State Delta、Evidence、Cost Event 报告中每个诊断结论都能回链到任务、轨迹、工具结果或专家标签。
Scoring Engine 组合程序化 verifier、LLM-as-judge、人类 rubric、pairwise preference、安全检查、成本效率、稳定性和生产指标；支持 scenario-specific 权重。 Score、Metric、Judge、Reviewer、Calibration Set 能解释总分来源，并展示评分一致性、judge drift、人工仲裁结果。
Failure Intelligence 建立 Agent failure taxonomy：理解错误、计划错误、工具选择、参数错误、状态追踪、政策违规、恢复失败、幻觉、越权、成本失控。 Failure、Root Cause、Severity、Business Impact、Suggested Fix 失败聚类能直接生成改进 backlog 和数据采购短名单。
Expert Operations 专家提交难例、创建 verifier、标注轨迹、校准 judge、复核争议；平台管理专家等级、质量抽检、一致性、返工和奖金。 Expert、Submission、Review Queue、Consensus、Bounty 隐藏集和高价值任务持续增长，专家一致性和泄露风险可监控。
Data Product Marketplace 将失败类型映射到任务包、verifier 包、golden trajectory、preference data、安全红队、GUI grounding、policy dataset、RAG eval dataset。 Dataset SKU、Quote、Sample Preview、Delivery Spec 报告能解释“为什么买这个数据包”，并展示样例、预估影响和交付范围。
Agent Solution Marketplace 按行业、任务、预算、安全等级、延迟、部署方式推荐 Agent 或 Agent 方案商；支持客户私有榜与公开榜隔离。 Solution、Vendor、Scenario Rank、Fit Score、Procurement Note 用户从业务问题进入，能得到候选 Agent、风险边界和试点计划。
Governance & Trust 隐藏集隔离、任务泄露检测、模型版本声明、供应商 attest、审计日志、数据授权、PII 脱敏、沙盒安全、反作弊和法律审核。 Audit Log、Attestation、Data License、Leak Signal、Access Policy 排名、报告和数据交付经得起客户安全、法务和采购审查。

端到端工作流

终态平台的体验应该是一条飞轮，而不是多个分散功能页。每次评测都产生更好的任务、数据和治理规则。

业务建模

导入业务流程、政策、工具、真实约束和成功定义。

环境生成

创建 API、数据库、GUI、RAG、文件、用户模拟和 deterministic reset。

Agent 运行

并行运行多版本 Agent，记录完整轨迹和成本。

混合评分

组合 verifier、人类专家、LLM judge、偏好和风险检查。

诊断修复

聚类失败，生成工具、prompt、权限和训练数据建议。

持续回归

将高价值失败沉淀为隐藏集和生产 gate，进入下一轮。

核心数据模型

数据模型要服务可复现、可审计、可改进。最重要的不是 Task 或 Score，而是 Trace 与 Evidence 的完整链路。

Agent

参赛与上线主体

字段：owner、version、deploy mode、tool manifest、permission policy、model/provider、runtime、budget、risk profile、attestation。

Task

可执行业务难例

字段：scenario、instruction、environment seed、policy、allowed tools、success criteria、verifier、rubric、gold trace、hidden level。

Environment

可复现系统状态

字段：API schema、database seed、GUI state、mock services、RAG corpus、user simulator、reset script、data license。

Trace

诊断与训练核心资产

字段：observation、action、tool call、tool result、state delta、screenshot、video、latency、token、cost、policy event。

Failure

可行动问题单元

字段：taxonomy、severity、root cause、evidence link、business impact、fix path、dataset recommendation、owner。

Dataset SKU

商业转化单元

字段：data type、scenario coverage、sample count、expert level、delivery format、license、expected lift、price band。

评分信号与 Verifier 组合

没有一种 verifier 能覆盖所有 Agent 任务。可验证系统执行、主观判断、多轮交互、合规边界和真实用户体验必须混合。

信号 适用任务 实现方式 风险控制
程序化 verifier 订单状态、数据库修改、代码测试、文件生成、RAG 引用命中。 SQL/state diff、unit test、API assertion、snapshot comparison。 为每个 verifier 建反例集，防止“过 verifier 但业务错”。
轨迹评分 长程任务、工具链、GUI 操作、多系统切换。 工具顺序、参数正确性、状态追踪、恢复动作、重复调用率。 保留原始轨迹证据，区分环境异常与 Agent 错误。
人类 rubric 复杂判断、客服政策、专业咨询、创意质量。 专家标注、双人复核、冲突仲裁、一致性监控。 校准样本、盲评、专家等级、低一致性任务重写。
LLM-as-judge 可扩展初筛、解释质量、引用一致性、合规初审。 标准化 judge prompt、reference answer、chain-free rationale、judge ensemble。 用 human disagreement 队列校准，监控 judge drift。
Pairwise preference 体验、表达、开放式任务、非确定性输出。 匿名对战、Bradley-Terry/Elo、偏好理由采集。 不能替代任务成功率；需要按用户群、行业和语言分层。
生产在线 eval 上线后的质量漂移、真实用户失败、成本异常。 production trace sampling、online judge、事件告警、回归数据集沉淀。 隐私脱敏、客户隔离、采样授权、低延迟异步采集。

失败分类 Taxonomy

失败分类要可训练、可修复、可售卖。每个 failure type 都应映射到修复动作和数据产品。

F1

意图理解错误

误读用户目标、缺少澄清、忽略约束。

F2

计划错误

步骤顺序错误、过早结束、长程目标丢失。

F3

工具选择错误

不用工具、错用工具、工具切换失败。

F4

参数与状态错误

字段填错、状态未更新、数据库结果误读。

F5

政策合规错误

违反退款、权限、风控、专业建议边界。

F6

安全与越权

PII 泄露、高风险动作、提示注入、隐藏集窃取。

F7

多轮交互失败

用户模拟中失去上下文、不能协商或确认。

F8

恢复能力不足

工具报错、环境异常、信息缺失时无法自救。

F9

幻觉与证据缺失

编造事实、引用不匹配、无法回链证据。

F10

成本效率失控

循环调用、token 过高、延迟过长、重试过多。

数据产品与报告转化

这是相对 Abaka/LangSmith 类产品可做差异化的地方：把测评结果变成数据采购和 Agent 采购的短名单。

失败信号 推荐数据产品 交付物 报告中应如何表达
计划/工具调用失败集中 Golden Trajectory Pack 专家动作序列、工具参数、状态差异、可训练轨迹。 展示失败步骤热区，并说明补轨迹数据预计覆盖哪些任务族。
Verifier 缺失或不稳定 Task & Verifier Design Pack 任务规格、环境脚本、自动 verifier、rubric、反例集。 说明当前无法规模化回归的原因和 verifier 建设优先级。

关键风险与设计原则

Agent 测评平台的核心风险不是“功能不够”，而是评测不可信、报告不可追溯、环境不安全、数据无法授权、商业转化断裂。

高风险

排行榜污染

隐藏集泄露、公开题过拟合、供应商提交特殊版本都会破坏信任。必须强制版本声明、隐藏集隔离、异常检测和第三方审计。

高风险

沙盒与数据安全

Agent 代码、浏览器环境、客户数据和专家任务必须隔离。默认使用合成/脱敏数据，真实数据进入任务池需授权。

中风险

Verifier 误导

自动 verifier 可能奖励错误捷径。每个 verifier 要有反例集、人工抽检和业务 owner 审批。

中风险

专家供给不足

高质量难例和金标轨迹是稀缺资产。需要专家等级、奖励、质量仲裁和任务模板提升供给效率。

产品原则

证据先于总结

每条报告结论必须回链 evidence。LLM 可以生成解释，但不能成为唯一证据。

商业原则

诊断必须可购买

所有失败类型都要映射到一项改进动作：数据、工具、权限、prompt、Agent 供应商或上线 gate。
