2. 目标生产架构

前端

Console Web

React/Next.js 或 Vue/Nuxt，包含控制台、报告查看、任务市场和数据商店。

后端

Arena API

Agent 接入、任务管理、赛季管理、评分、报告、结算、权限。

运行

Sandbox Runner

Kubernetes Job、Firecracker/gVisor、Browser/GUI 环境、工具代理和日志采集。

组件职责建议技术
Identity & RBAC企业客户、开发者、专家、平台运营权限隔离。Auth0/Okta/自研 IAM，PostgreSQL RBAC。
Agent Registry保存 Agent 接入信息、工具声明、运行策略和版本。PostgreSQL + Secret Manager。
Task Marketplace难例提交、审核、版本、隐藏集分层、奖励。PostgreSQL + Object Storage。
Sandbox Runner执行 Agent、连接模拟工具、浏览器或桌面环境。Kubernetes, Docker, Playwright, VNC, gVisor/Firecracker。
Trace Store保存 observation、action、tool result、state delta、截图、录屏。S3/GCS + PostgreSQL metadata + ClickHouse events。
Scoring Service程序化 verifier、rubric、pairwise preference、成本计算。Python/FastAPI workers + queue。
Report Generator失败分类、根因分析、数据包推荐。LLM-assisted pipeline + deterministic templates。
Commerce Bridge数据产品短名单、报价、CRM 线索。CRM integration, CPQ, billing workflow。

3. Agent 接入协议

生产系统建议提供三种接入方式，逐步降低客户进入门槛。

方式适合用户关键要求
API Endpoint已有 Agent 服务的客户/开发者。统一请求/响应 schema、签名鉴权、超时和速率限制。
Container Image需要完整运行时依赖的团队。镜像扫描、资源限制、出网策略、工具代理注入。
Platform Builder没有成熟 Agent 的业务团队。模板、工具选择、guardrail、系统提示和测试集联动。

4. 任务与 Verifier 设计

任务不应只是 prompt，而应包含环境、工具、验收和金标轨迹。

字段说明
instruction用户可见任务说明，含业务目标与限制。
environment_seed数据库、网页、文件、GUI 状态、工具响应的初始化种子。
allowed_toolsAgent 可以调用的工具列表和权限范围。
verifier二元成功/失败、分项评分或专家 rubric。
gold_trajectory专家推荐动作序列，用于训练和诊断。
failure_tags预期容易失败的原因，如澄清不足、工具误读、越权动作。

5. 评测运行流程

  1. 赛季创建：选择场景、任务包、公开/隐藏比例、评分权重、奖金池。
  2. Agent 报名：提交接入方式、版本、工具声明、权限策略和预算上限。
  3. 沙盒预检：镜像扫描、API 可达性、工具 schema、权限和成本阈值检查。
  4. 任务抽样：对公开集、隐藏集、专家审计集分层抽样，避免任务泄露。
  5. 并行执行：runner 调用 Agent，记录每一步 observation、action、tool result、state delta。
  6. 评分合并：程序化 verifier、专家 rubric、偏好对战和成本指标合并为综合分。
  7. 审计与防作弊：隐藏集异常、重复答案、可疑流量、过拟合模式和人工抽查。
  8. 输出结果：排行榜、奖金、失败模式、报告和数据产品推荐。

6. 报告生成 Pipeline

阶段输入输出控制点
轨迹聚合Trace events, verifier results任务级结果表去重、脱敏、权限过滤。
失败分类失败轨迹、专家标签Failure taxonomy保留证据链接，不只给 LLM 总结。
根因分析失败分类、任务元数据、工具状态根因和影响估计区分事实、推断和估计。
数据匹配失败类型、客户行业、任务场景数据产品推荐与 Appen SKU 映射表绑定。
报告发布模板、图表、证据HTML/PDF/在线报告客户数据隔离和审批流。

7. 数据产品 SKU 建议

以下 SKU 参考 Appen 公开 Agentic AI 数据服务叙事，并结合 Arena 评测产生的失败证据做产品化映射。生产化时建议维护一张可配置的“失败类型 → 数据 SKU → 样例交付物 → 报价规则”映射表。

SKU内容触发购买信号
Task & Verifier Design Pack任务规格、环境、自动 verifier、rubric。客户缺少可验证任务集，评测无法规模化。
Golden Trajectory Pack专家动作轨迹、工具调用、状态证据。Agent 规划和工具调用错误明显。
Failure Taxonomy Review轨迹级失败根因标注。客户知道失败多，但不知道如何采集训练数据。
Pairwise Preference Data专家偏好对战数据和 judge 训练样本。任务质量难以用二元 verifier 判断。
High-Risk Action Red Team越权、PII、退款、数据导出的拒绝和确认样本。安全合规率低或企业客户关注风控。
GUI Grounding Dataset桌面/网页截图、坐标动作、验收状态。Agent 在浏览器或桌面环境中定位不稳定。

9. 验证策略

  • 前端：Playwright E2E 覆盖主流程、响应式布局和可访问性扫描。
  • Runner：用固定 seed 的 mock Agent 做回归，确保评分可复现。
  • Verifier：每个 verifier 配套反例、边界例和人工审计样本。
  • 报告：报告中的每个结论必须可回链到任务、轨迹或专家标签。
  • 安全：沙盒逃逸、数据泄露、提示注入和隐藏集污染做专项红队测试。
